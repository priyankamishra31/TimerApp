Migration & Governance Rollout

📅 Version: 1.0 👤 Owner: DevEx Platform Team ⏱ Timeline: 16 Weeks 📦 Repos: ~300 🔧 Tool: github-governance CLI + Backstage

📋Project Overview

Mission Statement
We are migrating approximately 300 repositories from our existing GitHub Enterprise Server (GHES) instance to GitHub EU. In GitHub EU, repository settings and standards are controlled by Backstage and platform automation rather than per-engineer manual configuration. Alongside the physical migration, we are rolling out automated governance across all repositories.

Context & Goals

GitHub EU is managed through Backstage as the central developer portal. New repos must be created from Backstage templates. TechDocs is the target for internal documentation (mkdocs-based).

The github-governance CLI is the single source of truth for what a governed repository looks like. It writes and audits governance files, is wired into Backstage so new repos are born governed, and runs in batch mode to migrate existing repos through governance PRs.

Two core policies enforced post-migration:

  • Notebook policy: .ipynb files are permitted, but no cell outputs or execution counts may ever be committed.
  • Data file policy: Real data files (CSV, Parquet, pickle, NumPy, databases, archives) are forbidden except for small, explicitly declared test fixtures under tests/data/ (≤ 1 MB per file).

What Changes: GHES vs GitHub EU

⚠️ Before (GHES)

  • Manual repo creation via GitHub UI
  • No enforced notebook output policy
  • No data file restrictions
  • Ad-hoc CI/CD configuration per repo
  • No central software catalog
  • Branch protection configured manually
  • No pre-commit hooks standard
  • Engineers commit data files accidentally
  • Webhook/integration configs vary by team
  • No drift detection

✅ After (GitHub EU + Governance)

  • All repos created from Backstage templates
  • Notebook outputs auto-stripped by pre-commit
  • Data files blocked by pre-commit + CI gate
  • Standard CI via reusable central workflow
  • Backstage software catalog (source of truth)
  • Branch protection enforced: governance-ci required
  • Standard pre-commit hooks across all repos
  • CI blocks any PR adding data outside allowlist
  • Webhooks inventoried and migrated
  • Weekly drift detection + Grafana dashboard

System Architecture

Architecture Overview
The system has 5 integrated layers: Engineer local (pre-commit hooks), GitHub EU (repository + branch protection), GitHub Actions CI (governance workflow), github-governance CLI (write/audit tool), and Backstage (software catalog + templates).

Data Flow: Engineer → Merge

ENGINEER (local)
  git commit → [pre-commit hooks] → blocked/fixed OR clean commit
  git push → PR opened on GitHub EU
        ↓
GITHUB EU — Repository contains (written by github-governance):
  ├── .pre-commit-config.yaml
  ├── .github/workflows/governance.yml  (calls central reusable wf)
  ├── .gitignore
  ├── catalog-info.yaml
  └── mkdocs.yml

  Branch Protection: governance-ci ✅ required check
        ↓
GITHUB ACTIONS CI
  governance.yml triggers on every PR
  ├── notebook-guard: FAIL if .ipynb has outputs/execution counts
  └── data-file-guard: FAIL if data file outside tests/data/ or > 1MB
        ↓
CENTRAL GOVERNANCE REPO (org/governance)
  ├── .github/workflows/governance-reusable.yml
  ├── spec/  (versioned canonical file templates)
  └── CHANGELOG.md
        ↓
MONITORING
  Weekly scheduled: github-governance audit --all
  ├── feeds Grafana dashboard
  ├── opens GitHub issues for drifted repos
  └── Slack alert to #devex-oncall if drift > threshold

New Repo Flow (Post-Migration Steady State)

1. Engineer clicks "Create Repo" in Backstage
2. Backstage runs template scaffolder
3. Scaffolder action calls: github-governance apply <new-repo>
   ├── writes .pre-commit-config.yaml
   ├── writes .github/workflows/governance.yml
   ├── writes .gitignore
   ├── writes catalog-info.yaml [governance/spec-version: 1.x]
   └── writes mkdocs.yml
4. Repo exists in GitHub EU with all governance files in initial commit
5. Branch protection applied: governance-ci is a required check
6. Backstage catalog ingests catalog-info.yaml → repo appears in catalog
7. Engineer opens first PR → CI governance check runs immediately

🌊Phase 3 — Bulk Migration: ~300 Repos in Waves (Weeks 6–14)

Objective
Migrate all remaining repos from GHES to GitHub EU using the validated wave approach. Apply governance to every repo. Sequence by risk: inactive repos first, complex/special repos last.

8.1 Wave Plan

🌊 Wave 1 — Tier 1 Inactive25–30 repos | Weeks 6–7
Low risk; validate automation at scale. No active contributors — safe to migrate without disruption.
🌊 Wave 2 — Tier 1 Remaining + Tier 2 Low-Activity30–35 repos | Weeks 7–8
Process now proven; increase pace. Low-activity repos with minimal developer impact.
🌊 Wave 3 — Tier 2 Standard Active45–50 repos | Weeks 8–9
Bulk of repos; full team communication required. Group by team — all a team's repos migrate together.
🌊 Wave 4 — Tier 2 Standard Active (Continuation)45–50 repos | Weeks 9–10
Remaining standard active repos. Same process as Wave 3.
🌊 Wave 5 — Tier 3 Complex40–50 repos | Weeks 10–12
Repos with violations in history. Extended merge window: 10 business days instead of 5. CI starts in warn mode and stays there until team confirms cleanup complete (or 30 days, whichever first).
🌊 Wave 6 — Tier 3 Complex (Remaining)40–50 repos | Weeks 12–13
Same handling as Wave 5. DevEx tracks open legacy violations as a separate dashboard metric.
🌊 Wave 7 — Tier 4 Special / Bespoke CI / MonoreposVariable | Weeks 13–14
Handled one at a time. DevEx schedules a 30-minute 1:1 with each repo owner. Review what makes repo non-standard. Determine if policy exception needed or if repo can be brought to standard with small adjustments. Apply governance manually (not via batch script). Validate with repo owner before enforcement cutover.

8.2 Wave Gating Criteria

Do NOT start the next wave until ALL of the following are met for the current wave
  • ≥ 95% of governance PRs merged (not just opened)
  • Zero unresolved P0 issues (CI completely broken, team blocked on shipping)
  • False positive rate < 5% for the wave
  • Machine-readable log shows < 2% failed status repos
  • Wave metrics reviewed by DevEx lead and sign-off given

8.3 Wave Execution Process (Detailed)

T–7 days: Team Notification

Send to each affected team's Slack channel and by email:

Template Notification Message
Subject: [Action Required] Your repos are migrating to GitHub EU on [DATE]

Your repos are moving to GitHub EU on [DATE]. Here's what's happening:
Repos affected: [list]

What will happen:
• Your repos will be migrated with full history intact.
• A governance PR will be opened automatically. Please review and merge within 5 business days.
• CI checks will initially warn (not block). After you merge the PR, they will block.

What you need to do before [DATE]:
• Optionally: run pre-commit install in your repos and test locally.
• Optionally: strip outputs from any open PRs with notebooks.

Docs: [TechDocs link]   Questions: #devex-platform or tag @devex-oncall

T–2 days: Pre-Migration Dry Run

github-governance apply --dry-run --repo-list wave-N.txt > wave-N-dryrun.json

DevEx engineer reviews output: repos with unexpectedly large diffs → move to next wave. Repos with unresolvable conflicts → flag for Wave 7 manual handling. All others → confirm as ready.

T+0: Migration Day

Run sequentially per repo (can be parallelised per team, but not all 50 at once to avoid CI queue saturation):

  1. Trigger GH Enterprise Importer for the repo
  2. Wait for import to complete (poll until status = complete)
  3. Verify: git log --oneline -5 on GHES and GitHub EU match
  4. Apply branch protection rules (warn mode, governance-ci required but not yet blocking)
  5. Run github-governance apply <repo> — opens governance PR
  6. Log outcome to wave-N-results.json

Entire wave should complete within one business day. If any repo fails import, skip it, flag it in the log, and add it to the next wave.

T+1 to T+5: Merge Window

  • DevEx monitors PR merge rate daily
  • CI runs in warn mode (governance.yml mode: warn)
  • Any team that reports a false positive gets an immediate response from DevEx (same business day)
  • At T+5: any unmerged governance PRs are merged by DevEx (pre-communicated policy, announced at T–7)

T+5: Enforcement Cutover

Switch mode: warn → mode: block for all wave repos in one commit to the central governance repo. Send Slack notification to affected teams.

T+7: Wave Close Review

DevEx engineer reviews wave-N-results.json: stats (applied/skipped/failed/false positives, PRs auto-merged vs team-merged), patterns requiring spec or tooling updates, sign off on wave completion.

Within 2 weeks of Wave 7 completing:

github-governance audit --all --format json > coverage-audit.json

Remediate all gaps. This is the final gate before declaring migration complete.

9.2 Scheduled Drift Detection

Set up a weekly scheduled GitHub Actions job in the central governance repo:

Schedule: Every Monday at 09:00 UTC

Job steps:
1. Query Backstage catalog API for all active repos
2. Run github-governance audit --all --format json
3. Parse results:
   - Repos with status: drift or status: missing → open GitHub issue with repo list
   - Repos with spec version > 1 minor version behind → add to "upgrade queue" issue
4. If drift_count > 5: post to #devex-oncall Slack channel
5. Write metrics to Grafana via push gateway

3.4 Exception Process

  1. Team submits an exception request via the Backstage governance exception form.
  2. DevEx reviews within 3 business days.
  3. If approved, the exception is written into the repo's catalog-info.yaml under governance/exceptions.
  4. github-governance apply and audit respect declared exceptions.
  5. Exceptions are reviewed quarterly. Expired exceptions are removed automatically.

12.2 Rollback: Single Repo (Target: < 30 minutes)

Trigger: Team reports critical regression (CI blocks a legitimate operation, pre-commit causes breaking errors).

Runbook:

  1. DevEx engineer clicks "Revert" on the governance PR in GitHub EU
  2. The governance CI workflow is now absent; branch protection falls back to no governance-ci requirement
  3. Repo is effectively ungoverned
  4. DevEx logs the incident, diagnoses root cause
  5. If root cause is a false positive in a hook/check → fix in central workflow, re-test
  6. If root cause is a policy edge case → add exception to catalog-info.yaml
  7. Re-open governance PR after fix; repo goes back through the 5-day merge window

12.3 Rollback: Wave-Level (Target: < 2 hours)

Trigger: More than 3 repos in the same wave report the same critical regression.

Runbook:

  1. DevEx lead calls wave halt immediately
  2. In the central governance repo, change the reusable workflow to mode: warn for all repos (single commit, propagates automatically)
  3. Repos stop blocking PRs within minutes (no per-repo action required)
  4. Convene 30-minute incident review within 24 hours
  5. Identify and fix root cause
  6. Update spec if needed (increment patch version)
  7. Re-run wave dry-run against affected repos before restarting

12.4 Handling Legacy Violations Safely

CI only scans the PR diff, not the full repository tree
  • A repo with bad notebooks or data files committed years ago does NOT have its CI fail on every PR — only if the PR itself touches those files.
  • This means governance can be applied to 100% of repos without immediately breaking teams working around legacy debt.
  • Teams clean forward at their own pace; DevEx tracks progress via dashboard.

Pros & Cons Summary

✅ Pros of This Migration

  • All repos governed consistently from day one
  • Automated enforcement — no manual policing
  • Backstage as single pane of glass for all repos
  • New repos born governed (zero manual steps)
  • CI catches notebook/data violations before they accumulate
  • Drift detection catches config rot weekly
  • GHES 90-day read-only window = safe fallback
  • Wave-gated approach limits blast radius if something breaks
  • Rollback runbooks tested and documented
  • Full git history preserved by GH Enterprise Importer

⚠️ Cons / Trade-offs

  • 16-week timeline requires sustained DevEx resource commitment
  • Pre-commit hooks add friction for engineers used to committing freely
  • CI false positives (e.g., legitimate small CSV) will happen and require DevEx response
  • Tier 3 legacy violations require team cleanup effort
  • Tier 4 / bespoke repos need manual handling — can't be fully automated
  • Webhook / integration re-wiring may surface hidden dependencies
  • GitHub Packages namespace change requires team pipeline updates
  • Policy exceptions add ongoing governance overhead
